Part 2 — Лаб 2: Жишээ систем (eSIM худалдаа) + SDLC шаардлага + Use-case тест

Жишээ систем: eSIM online худалдаа SDLC: Requirements → Use-case → Test RFP ↔ Requirements нийцүүлэлт
Зорилго: RFP дээр үндэслэн шаардлага тодорхойлж (FR/NFR), use-case diagram гарган, гол урсгалуудыг test case-ээр туршиж шалгасан байдлаар тайлбарлах.

Executive Summary

Товч танилцуулга

  • Хэрэглэгч eSIM багц хайж сонгоно → захиалга үүсгэнэ → төлбөр төлнө → төлбөр баталгаажна → eSIM activation/QR авна.
  • Системийн зорилго: худалдан авалтын алхмыг хурдан, ойлгомжтой, найдвартай болгох.
  • Интеграц: Payment Gateway + eSIM Provider API + Email Service.

Яагаад хэрэгтэй вэ? Гар ажиллагааг багасгаж, төлбөр/идэвхжүүлэлтийн мэдээлэл алдагдах эрсдэлийг бууруулна.

Deliverables: Requirements (FR/NFR), Use-case diagram, Test cases, Traceability

Scope (хүрээ)

  • Орох: Plan browse, order create, payment, verify, activation fetch, email send, usage check.
  • Орохгүй (if needed): Refund, loyalty, full CRM (энэ лабораторид заавал биш).
Нэмэлт тэмдэглэл
Scope-оос гаргасан функцуудыг багш асуувал “Lab хүрээнд үндсэн урсгал дээр төвлөрсөн” гэж тайлбарлана.

Stakeholders & Actors

Actor list

Хэрэглэгч: plan хайх/үзэх, захиалга үүсгэх, төлбөр хийх, QR/activation авах, usage шалгах
Админ: plan удирдах, захиалга хянах/удирдах
Payment Gateway: төлбөрийн invoice + verify/webhook
eSIM Provider API: activation (QR) мэдээлэл өгөх
Email Service: activation мэдээллийг имэйлээр илгээх
Акторын харилцаа
Хэрэглэгч төлбөр төлөх үед gateway-тай харилцана. Төлбөр баталгаажмагц provider API-аас QR авч, email service-р дамжуулан хэрэглэгчид илгээнэ.

RFP Summary (Захиалагчийн хүсэлт)

RFP-ийн гол шаардлага (товч)

  • Хэрэглэгч eSIM багцыг онлайнаар сонгон худалдан авах.
  • Payment gateway ашиглан төлбөр хийх ба төлбөрийн статус баталгаажуулах.
  • Төлбөр баталгаажсаны дараа eSIM activation/QR мэдээллийг хэрэглэгчид хүргэх.
  • Админ талд багц болон захиалгыг удирдах.
Нийцүүлэлт: Доорх FR/NFR болон use-case/test case-үүд нь дээрх RFP-ийн мөр бүрийг хангана.

Requirements (FR / NFR)

Functional requirements

  • FR1 — Plan хайх/үзэх
  • FR2 — Plan дэлгэрэнгүй харах
  • FR3 — Захиалга үүсгэх (Create order)
  • FR4 — Төлбөр хийх (Make payment)
  • FR5 — Төлбөр баталгаажуулах (Verify payment)
  • FR6 — eSIM activation (QR) авах (Provider API)
  • FR7 — Имэйл илгээх (Send activation email)
  • FR8 — Usage шалгах
  • FR9 — Admin: Manage plans / Manage orders

Non-functional requirements

  • NFR1 — Аюулгүй байдал: webhook/verify request validation (signature/token)
  • NFR2 — Гүйцэтгэл: invoice/QR хурдан харагдах
  • NFR3 — Найдвартай ажиллагаа: retry + error handling
  • NFR4 — Ашиглахад хялбар: алхамууд ойлгомжтой UX
Acceptance criteria (товч)
Жишээ: FR4/FR5 — Хэрэглэгч төлбөр төлсний дараа order статус PAID болж, activation харагдаж, имэйл ирнэ.

Use-Case Testing (Test Cases)

Test cases (богино, лабораторид хангалттай)

TC1 — Амжилттай төлбөр
Алхам: Plan сонгох → Order үүсгэх → QR гаргах → Төлбөр төлөх → Verify/ webhook → Provider API → Email
Expected: Order = PAID, activation/QR харагдана, имэйл явна.
TC2 — Төлбөр амжилтгүй
Алхам: QR гаргах → Төлбөр амжилтгүй/цуцлах → Verify хийх
Expected: Order PAID болохгүй, хэрэглэгчид retry/өөр төлбөрийн арга санал болгоно.
TC3 — Provider API timeout
Алхам: Order PAID → Provider API дуудлага timeout
Expected: Retry хийнэ; амжилтгүй хэвээр бол “түр хүлээнэ үү” мессеж + activation pending.
TC4 — Email илгээхэд алдаа
Алхам: Activation авна → Email service алдаа буцаана
Expected: Сайт дээр activation харагдана; дахин илгээх боломжтой.
TC5 — Usage шалгах буруу код
Алхам: Буруу код оруулах
Expected: “Олдсонгүй/алдаа” гэсэн ойлгомжтой хариу.
Яагаад тест хэрэгтэй вэ? Use-case дээр тодорхойлсон урсгал бодит нөхцөлд зөв ажиллаж буйг нотолно.

  • Гол урсгалуудыг test case-ээр шалгаж баталгаажууллаа (амжилттай төлбөр, алдааны нөхцөлүүд).
